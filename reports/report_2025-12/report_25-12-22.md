## 1. Executive Summary

This report reviews the current state of QuFLX v2 after the recent Gateway / refresh-assets fixes and stream status improvements, with a focus on:

- Robustness of the Get Assets flow (FavoriteStarSelect + `/api/v1/refresh-assets` + Dashboard).
- Accuracy and stability of status indicators and the Live Feed badge.
- Potential fragilities or inconsistencies that could cause the UI to misrepresent system health.

Overall:

- The pipeline **Collector → Redis → Gateway → Dashboard** is structurally sound and aligns with CORE_PRINCIPLES.
- The latest fixes have removed the critical 500-refresh bug and made stream health indicators tick-driven (much closer to user expectations).
- Remaining concerns are mostly around **semantics and edge cases**, not hard bugs: how we interpret status, OTC-only selection, and a few duplications in status polling.

## 3. Code Review – Key Areas

### 3.1 Stream Health & Live Feed Badge

**Files reviewed:**

- `backend/services/gateway/main.py` (Redis listener, `system_state` updates).
- `gui/Dashboard/src/store/marketStore.js` (socket handlers, `lastTickTimestamp`).
- `gui/Dashboard/src/hooks/useStreamHealth.js`.
- `gui/Dashboard/src/components/ChartWorkspace.jsx` and `TopBar.jsx`.

**Findings (Reviewer + Frontend-Specialist):**

- `lastTickTimestamp` is updated on every valid tick in `marketStore.js:388–415` and fed into `useStreamHealth`.
- `useStreamHealth` now derives health *purely* from tick recency (idle / streaming / slow / stale). This is simple, predictable, and matches your observed behavior (badge turns green once ticks arrive).
- The chart-level badge and the TopBar Stream badge both consume the same hook, so they stay consistent.

**Potential fragilities / semantic concerns:**

- If the Collector stops but the last tick is recent (e.g., user pauses for a few seconds), the Stream badge will still show `streaming` until `diff > 5s`. This is acceptable and expected, but it means "Stream" really means "ticks are flowing recently", not "collector service is healthy".
- Backend `system_state.stream` is still maintained (based on collector `SystemStatus` messages) but is no longer the main driver for UI health. This is a design choice; it should be documented so future changes don’t try to rewire it inconsistently.

**Assessment:**

- From a robustness standpoint, the tick-driven health is preferable to the previous collector-status-only approach.
- No infinite loops or unstable dependencies were found in `useStreamHealth`.

### 3.2 Status Indicators & Polling

**Files reviewed:**

- `gui/Dashboard/src/store/marketStore.js` (status polling + socket handlers).
- `gui/Dashboard/src/components/StatusIndicator.jsx`.

**Findings (Reviewer + Frontend-Specialist):**

- There are two status sources:
  1. `/api/v1/status` polled every 30 seconds via `fetchStatus()` invoked in `connectSocket()`.
  2. Socket-based `check_status` / `backend_status` messages driven by `StatusIndicator` and the `checkBackendStatus` action.
- `StatusIndicator` effect now:
  - Triggers an immediate `checkBackendStatus()` when the socket becomes available.
  - Schedules a `setInterval` (5s) that calls `checkBackendStatus()` until unmounted.
- `backend_status` handler updates both `backendStatus` and `chromeStatus` in store.

**Potential fragilities:**

- **Duplication:** There is some overlap between `fetchStatus` and `checkBackendStatus` (both ask the backend for status). Currently this is safe, just redundant.
- **Polling frequency:** 5-second interval is fine, but combined with 30-second `/status` polling it may be slightly higher than strictly necessary.

**Assessment:**

- Logic is stable and easy to reason about. Any future bugs here are more likely to be semantic (what we show) rather than structural (app breaking).

### 3.3 FavoriteStarSelect & Refresh Assets

**Files reviewed:**

- `capabilities_v2/favorite_star_select.py`.
- `capabilities_v2/runner.py`.
- `backend/services/gateway/main.py` (`/api/v1/refresh-assets`).
- `gui/Dashboard/src/store/marketStore.js` (`refreshAssets` action).

**Findings (Reviewer + Backend-Specialist):**

- The capability runner uses a consistent pattern: `{"ok": true/false, "data": {...}, "error": ...}`.
- `_parse_script_json` in the Gateway correctly handles the mixed output (status line + JSON) from the runner.
- `FavoriteStarSelect`:
  - Properly reads `min_pct`, `sweep_all`, `unstar_below`, `max_assets`, and `target_assets`.
  - Propagates `max_assets` and `target_assets` into `_process_entire_list` and `_process_visible_only`.
  - Enforces `max_assets` as cap on `star_clicked` count, recording skipped assets under `skipped_max_limit`.
  - Applies `target_assets` only when non-empty, using normalized symbol comparison.
- The Gateway endpoint builds inputs correctly and returns a clean `assets` list plus metadata used by the Dashboard.

**Potential fragilities:**

- No built-in OTC-only filter yet. Filtering OTC vs non-OTC is currently implicit (by which symbols appear and which you target manually).
- `assetFilterState` lives in the same store as everything else; it works fine now but reinforces the observation from earlier reports that `marketStore.js` is doing a lot.

**Assessment:**

- The Get Assets flow is now structurally robust. Main remaining improvement is feature-level: an OTC-only filter, not a bug.

## 4. New Feature Notes (Non-Blocking)

### 4.1 OTC-Only Filter for Get Assets

**Current:**

- The Get Assets flow can be configured by Max Assets and Specific Assets.
- OTC vs global FX distinction is not enforced; some lower-payout or non-OTC pairs may be included depending on Pocket Option’s current offerings.

**Desired:**

- A toggle or filter mode such that only OTC instruments are starred (and listed) while standard FX pairs are ignored.

**Approach (high level):**

- Add an optional `filter_mode` input to `refresh-assets` payload (e.g., `'otc' | 'fx' | null`).
- Extend `FavoriteStarSelect` to interpret `filter_mode` and skip assets based on suffix/normalized label.
- Expose this in the Dashboard as a simple checkbox or toggle in the 92% panel.

### 4.2 Workflow Text → Tooltip

**Current:**

- The 92% Payout Assets panel includes an instructional text block at the top:
  - “Workflow: Set max assets & specific targets below, then click "Get Assets" to star them. Manually select in Pocket Option UI.”

**Desired:**

- Replace the static text with a hover tooltip attached to an info icon or label, to reduce visual clutter while still keeping the workflow clear.

**Approach (high level):**

- Wrap a small info icon or label with a tooltip component (can be implemented with Tailwind + a simple absolute-positioned div; no external UI libs).
- Move the instructional text into that tooltip.

These are UX-level improvements and do not affect core robustness.

## 5. Recommended Action Plan

### Priority 1 – Clarify and Lock-In Stream Semantics

1. **Document tick-driven stream health**
   - In `docs` or a short section in the status documentation, clarify that:
     - `Live Feed` and `Stream` badges are tick-driven using `lastTickTimestamp`.
     - Collector status (`collector` field) is a separate concept and may be exposed as its own badge in future.
   - This prevents future regressions where someone tries to wire stream health back to collector status only.

2. **(Optional) Add a separate Collector badge**
   - If desired, introduce a dedicated “Collector” badge in the TopBar that maps directly to `system_state.collector`.
   - This keeps “Stream” focused on tick flow and avoids overloading a single indicator.

### Priority 2 – OTC Filter & Workflow UX

3. **Implement OTC-only filter in FavoriteStarSelect & Dashboard**
   - Add a `filter_mode` parameter (`'otc' | 'fx' | null`) to:
     - Dashboard `refreshAssets` payload.
     - Gateway `/api/v1/refresh-assets` endpoint.
     - `FavoriteStarSelect.run()` and its internal filtering logic.
   - Ensure this filter is applied *before* payout checks, so only desired classes of assets are considered.

4. **Convert 92% panel workflow text into a tooltip**
   - Replace the static workflow sentence with an info icon + tooltip in `AssetPanel.jsx`.
   - Keep the message identical to preserve current guidance.

### Priority 3 – Status Polling Consolidation

5. **Consolidate status polling strategy**
   - Decide whether `/api/v1/status` polling is still needed every 30s now that `backend_status` exists.
   - If not strictly needed, consider:
     - Reducing the frequency, or
     - Making `/api/v1/status` a manual debug endpoint only.
   - This reduces background traffic noise and simplifies mental model.

### Priority 4 – Long-Term Store Structure

6. **Revisit `marketStore.js` splitting** (already identified in earlier reports)
   - When workload allows, split into separate hooks/stores:
     - Connection/Status store (socket, wsStatus, chromeStatus, backendStatus).
     - Market & history store (selected asset/timeframe, historyCandles, payoutAssets).
     - Ticker store (marketData, quotesByAssetKey, baselineByAssetKey).
   - This is not urgent today but will keep growth manageable and aligned with CORE_PRINCIPLES (strict separation of concerns).

## 6. CORE_PRINCIPLES Alignment (Reviewer Summary)

- **Functional Simplicity:**
  - Recent changes simplified stream health to a single tick-driven hook, avoiding over-complex status logic.

- **Sequential Logic:**
  - The status and refresh flows follow clear steps: capability → Gateway → REST/Socket → store → components.

- **Incremental Testing:**
  - Lint passes and manual end-to-end flows (startup, select asset, Get Assets, stream) have been verified.

- **Zero Assumptions:**
  - `_parse_script_json` removed the assumption of pure-JSON stdout from capabilities.

- **Defensive Error Handling:**
  - Capability parsing failures now log errors with raw output included.

- **Separation of Concerns:**
  - Still some concentration in `marketStore.js`, but slices and hooks are moving in the right direction.

No critical structural blockers were identified for production-style use based on the current session. The recommendations above focus on tightening semantics, UX, and long-term maintainability rather than urgent bug fixes.
