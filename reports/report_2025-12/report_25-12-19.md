# Engineering Report: QuFLX v2 Status & Review
**Date:** 2025-12-19
**Author:** @Reviewer (delegated by @Team-Leader)

## 1. Executive Summary
This report reviews QuFLX v2 as of 2025-12-19 with focus on:

- Recent improvements in the Dashboard (charting, asset selection, OTC ticker panel).
- Backend streaming path from Collector тЖТ Redis тЖТ Gateway тЖТ Frontend.
- Code quality and adherence to `/c:/QuFLX/v2/.agents/CORE_PRINCIPLES.md`.

Overall, the system has made strong progress towards a clean, modular architecture. The Dashboard is significantly more coherent, state management is centralized in `marketStore.js`, and chart behavior now matches intraday expectations. There are still areas where simplification, validation, and explicit error handling will pay off.

## 2. Current Status Snapshot

### 2.1 Frontend (Dashboard)

**Strengths**
- Modular layout: `Dashboard.jsx` orchestrates `Sidebar`, `TopBar`, `AssetPanel`, and `ChartWorkspace` cleanly.
- State management: `gui/Dashboard/src/store/marketStore.js` centralizes WS status, selected asset/timeframe, history, and payout assets.
- Chart correctness:
  - Intraday candles now use UNIX timestamps in seconds and render correctly on the time axis.
  - Tick aggregation for multiple timeframes (1m, 5m, 15m, 1h) is implemented and wired to `selectedTimeframe`.
- OTC ticker panel:
  - Panel supports List vs Ticker modes.
  - Ticker mode builds a vertical OTC quote list from live `market_data` events.
- Lint/build health:
  - `npm run lint` and `npm run build` in `gui/Dashboard` both pass.

**Issues / Risks**
- **MEDIUM тАУ Implicit data contracts:**
  - `marketStore.js` assumes `market_data` payloads expose `asset`, `price`, `open`, `close`, `timestamp` consistently. These contracts are not yet formalized at the type/validation layer, which can lead to fragile behavior if backend payloads change.
- **LOW тАУ ChartWorkspace complexity:**
  - `ChartWorkspace.jsx` is starting to accumulate multiple responsibilities: chart init, history loading, tick aggregation, indicator management, UI controls, and WS status badge logic.

### 2.2 Backend (Gateway & Collector Integration)

**Strengths**
- Async Redis listener in `backend/services/gateway/main.py` correctly:
  - Subscribes to `market_data`, `trading:signals`, `system_status`.
  - Emits `market_data` events to asset-scoped rooms `market_data:{asset}`.
- Frontend `marketStore.js` uses `socket.io-client` to:
  - Connect and track `wsStatus` (connected/disconnected/error).
  - Subscribe to the selected asset room(s) and emit `select_asset` to the gateway.
- Status endpoints:
  - `/api/v1/status` surfaces collector/stream status that is displayed in `TopBar`.

**Issues / Risks**
- **MEDIUM тАУ Stream status semantics:**
  - `streamStatus` is currently derived from collector connectivity rather than explicit тАЬstream is actively pushing ticksтАЭ. This can cause the UI to show тАЬLive FeedтАЭ as streaming even if ticks are temporarily idle.
- **LOW тАУ Gateway cohesion:**
  - The gateway still combines HTTP API, Socket.IO, Redis listener, and Selenium orchestration in a single module. This is manageable now but should be watched to avoid future tangling.

### 2.3 Error Handling & Observability

**Strengths**
- Many previously silent `except` blocks in the collector and Selenium control paths were removed or replaced with logging.
- Dashboard shows a dedicated error banner in `ChartWorkspace.jsx` when `lastError` is set.

**Issues / Risks**
- **MEDIUM тАУ Inconsistent error surfacing:**
  - Some backend endpoints and event handlers still log errors without returning structured feedback to the frontend (or callers), which can make UX inconsistent.
- **LOW тАУ Logging granularity:**
  - There is still a risk that high-frequency paths could be noisy if logging levels are misconfigured (e.g., verbose per-tick logs in production).

## 3. Code Review тАУ Highlights

### 3.1 Dashboard Store (`marketStore.js`)

**Strengths**
- Clear separation of UI state (sidebar, tabs), market state (selected asset/timeframe, history, payout assets), and connection state (socket, wsStatus, chromeStatus, streamStatus).
- Socket connection logic unifies:
  - Status polling via `/api/v1/status`.
  - Room subscription management via `subscribedAssetKeys`.
  - Event handling for `market_data`, `system_status`, and asset selection results.
- History bootstrap logic uses a primary `/api/v1/bootstrap-history` path and a fallback `/api/v1/history/{asset}`.

**Concerns**
- **MEDIUM тАУ Single-store responsibility creep:**
  - `marketStore.js` is handling:
    - Socket lifecycle and room management.
    - History loading.
    - Ticker quote management (`quotesByAssetKey`, `baselineByAssetKey`).
    - Auto-refresh, status polling, and history collection triggers.
  - This is still manageable, but further growth would benefit from splitting by concern (e.g., `useConnectionStore`, `useMarketDataStore`) to maintain clarity.

### 3.2 Chart Workspace (`ChartWorkspace.jsx`)

**Strengths**
- Correct use of `createChart` and `CandlestickSeries` from Lightweight Charts.
- Clean tick aggregation logic:
  - Maps `selectedTimeframe` to an interval in seconds (1m, 5m, 15m, 1h, 4h).
  - Buckets ticks into candles and updates using `update()`.
- UX polish:
  - Loading overlay when switching assets or timeframes.
  - Indicator management and error banner are integrated.
  - тАЬLive FeedтАЭ badge bound to `streamStatus` with a visible green pulse when streaming.

**Concerns**
- **MEDIUM тАУ Multi-responsibility component:**
  - The component currently handles:
    - Chart creation and teardown.
    - History mapping and setting.
    - Live aggregation.
    - Indicator UI.
    - Top control bar logic.
    - Stream status visual.
  - According to CORE_PRINCIPLES (strict separation of concerns), a future refactor into smaller components/hooks (e.g., `useChart`, `useTickAggregation`, `ChartHeader`, `ChartStatusBadge`) would improve maintainability.

### 3.3 Asset Panel (`AssetPanel.jsx`)

**Strengths**
- Data source controls (Upload CSV, Live Feed, Get Assets, Collect History) are clearly presented.
- 92% Payout list is scrollable and wired to `setSelectedAsset`.
- OTC ticker mode:
  - Uses `TickerTape` to show a vertical list of assets with last price and % change.
  - Reuses `quotesByAssetKey` and `tickerMaxAssets` from the store.

**Concerns**
- **LOW тАУ Implicit assumptions about payout assets:**
  - The panel assumes `payoutAssets` is reliably populated; if backend refresh fails silently, the panel can appear empty without context. Handling of this case is deferred to the storeтАЩs error handling.

## 4. Areas for Optimization & Simplification

### 4.1 Simplify and Segment Frontend Logic

- **ChartWorkspace.jsx:**
  - Extract a dedicated hook for chart lifecycle, e.g. `useCandlestickChart(containerRef)` that only knows about series creation, resizing, and teardown.
  - Extract tick aggregation into a separate hook that consumes `marketData[selectedAssetKey]` and pushes updates to the chart instance.
  - Move indicator UI into a child component to reduce JSX and keep responsibilities focused.

- **marketStore.js:**
  - Consider a future split into sub-stores or slices:
    - Connection slice: socket, wsStatus, chromeStatus, streamStatus, fetchStatus.
    - Market slice: selectedAsset, timeframe, history, marketData.
    - Ticker slice: payoutAssets, tickerMaxAssets, quotesByAssetKey, baselineByAssetKey.
  - This is not urgent but will make future feature work safer.

### 4.2 Strengthen Data Contracts & Validation

- Introduce explicit schemas (Pydantic in backend, lightweight TS typing or runtime guards in frontend) for:
  - `market_data` payloads.
  - `bootstrap-history` and `history` responses.
- Validate parsed fields (asset, price, timestamps) early and fail fast with logged errors + clear frontend messages, aligning with CORE_PRINCIPLES (Fail Fast, Fail Loud).

### 4.3 Clarify Stream Status Semantics

- Distinguish between:
  - **Collector connected but idle** vs **actively receiving ticks**.
- Potential improvement:
  - Gateway can maintain a short-lived тАЬlast tick received atтАЭ timestamp and expose a richer `/status` payload, e.g. `{ collector: 'connected', stream: 'streaming', last_tick_age_ms: 120 }`.
  - Frontend can then decide when to show Live Feed as active vs degraded.

## 5. Proposed Action Plan

**Priority 1 тАУ Data Contracts & Validation**
- Define and document the JSON shape of `market_data`, `bootstrap-history`, and `history` responses.
- Add validation at the gateway level and defensive checks in `marketStore.js` to avoid undefined fields causing subtle UI issues.

**Priority 2 тАУ Frontend Separation of Concerns**
- Refactor `ChartWorkspace.jsx` into:
  - A chart container component focused solely on chart lifecycle.
  - Hooks for history loading and tick aggregation.
  - A separate component for indicators and the header controls.
- Keep behavior identical while reducing per-file complexity.

**Priority 3 тАУ Connection & Stream Status Clarity**
- Extend `/api/v1/status` to include:
  - Last tick timestamp and a derived health level (e.g., `stream: 'streaming' | 'idle' | 'stale'`).
- Update `TopBar` and the chartтАЩs Live Feed badge to reflect these more granular states.

**Priority 4 тАУ Store Structure Hardening (Medium-Term)**
- Plan a gradual reorganization of `marketStore.js` into smaller, purpose-focused slices while preserving the public API used by components.
- Review auto-refresh and history collection flows for opportunities to simplify or centralize timer logic.

**Priority 5 тАУ Documentation & Developer Onboarding**
- Update in-repo docs to reflect the current architecture:
  - Collector тЖТ Redis тЖТ Gateway тЖТ Dashboard pipeline.
  - How `marketStore.js` subscribes to rooms and filters data.
  - Conventions for asset keys and timeframes.

These steps will improve maintainability, reduce risk as features evolve, and ensure continued alignment with the CORE_PRINCIPLES (functional simplicity, separation of concerns, fail-loud error handling).
