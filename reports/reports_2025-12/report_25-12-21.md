# QuFLX v2 Status & Refresh Assets Fix Report (2025-12-22)

## 1. Observed Symptoms

- Dashboard status after restart:
  - Chrome Connection indicator: Yellow
  - Stream Connection indicator: Yellow
  - WebSocket (WS) indicator: Green
  - Chart top-left indicator: Offline
- Gateway logs while Dashboard is running:
  - `/api/v1/status` returning 200 OK with `redis_connected=True`, `socket_io_ready=True`, `chrome_debugging_available=True`, `ready_for_assets=True`.
  - `system_state` shows `collector='disconnected'` and `stream='idle'`, but with non-zero `last_tick_ts` and `last_tick_asset='AEDCNYOTC'`.
- When clicking **Get Assets**:
  - Browser console: `POST http://localhost:8000/api/v1/refresh-assets 500 (Internal Server Error)`.
  - Gateway log error:
    - `Invalid JSON output: âœ… Attached to Chrome session: https://pocketoption.com/en/cabinet/demo-quick-high-low` followed by the JSON payload.
    - `Refresh assets failed: 500: Invalid script output`.

## 2. Root Cause Analysis

### 2.1 Capability Runner Output Format

- The shared capability runner `capabilities_v2/runner.py` prints a human-friendly status line *before* emitting JSON:
  - Example for `refresh_assets`:
    - `✅ Attached to Chrome session: https://pocketoption.com/en/cabinet/demo-quick-high-low`
    - `{"ok": true, "data": { ... }}`
- Backend endpoints in `backend/services/gateway/main.py` assumed `stdout` was **pure JSON** and called `json.loads(result.stdout)` directly.
- For `refresh_assets`, this caused a `JSONDecodeError`, logged as:
  - `Invalid JSON output: âœ… Attached to Chrome session: https://pocketoption.com/en/cabinet/demo-quick-high-low`.
- The error was then wrapped as:
  - `HTTP 500: Invalid script output`.
- Result: **`/api/v1/refresh-assets` always returned 500**, and the Dashboard asset list never updated even though stars were clicked successfully in Pocket Option.

### 2.2 Affected Endpoints

- `backend/services/gateway/main.py`:
  - `/api/v1/refresh-assets`
  - `/api/v1/bootstrap-history`
  - `/api/v1/select-asset`
- All three parsed `result.stdout` with `json.loads` and would fail if any non-JSON text preceded the JSON block.

## 3. Implemented Fixes

### 3.1 Robust JSON Parsing Helper

- Added `_parse_script_json(stdout: str) -> Dict[str, Any]` to `backend/services/gateway/main.py`:
  - Strips empty lines.
  - Scans from the bottom up for a line starting with `{` or `[` and attempts `json.loads` on that line.
  - If not found, falls back to searching for the first `{` in the entire `stdout` and attempts `json.loads` from that index.
  - Raises a clear `ValueError` if no JSON fragment can be found.
- Purpose: tolerate informational lines (like the Chrome attach message) while still enforcing that the core payload is valid JSON.

### 3.2 Endpoint Updates

- `/api/v1/refresh-assets` now uses `_parse_script_json`:
  - Safely extracts the `{"ok": true, ...}` JSON block from the runner output.
  - On success, maps `data.processed.selected_now` and `data.processed.already_favorited` into a cleaned `assets` list.
  - On failure, logs `Invalid JSON output from refresh_assets: ... | raw=...` and returns `HTTP 500: Invalid script output`.
- `/api/v1/bootstrap-history`:
  - Replaced direct `json.loads(result.stdout)` with `_parse_script_json` to handle any future non-JSON prefix lines.
- `/api/v1/select-asset`:
  - Replaced `json.loads(result.stdout)` with `_parse_script_json` to avoid brittle behavior if the underlying script ever prints additional status lines.
- All changes compiled successfully via `python -m py_compile backend/services/gateway/main.py`.

   - Gateway log should show no new `Invalid JSON output` errors for refresh-assets.

### 5.2 Short-Term Hardening

1. Add lightweight logging in the Dashboard to surface backend errors in the UI (toast or banner) when `/api/v1/refresh-assets` returns non-200.
2. Extend the status indicators so the chart offline state also considers `last_tick_ts` age, not only `collector` status, to better reflect “historical only” vs “live streaming”.
3. Add a small unit-style test or script to exercise `_parse_script_json` with:
   - Plain JSON.
   - Prefixed status line + JSON.
   - No JSON at all (verify it fails loudly).

### 5.3 Medium-Term Improvements

1. Standardize runner output for all capabilities:
   - Option A: Move human-friendly log lines to `stderr` and reserve `stdout` for JSON only.
   - Option B: Gate the checkmark line behind a `--human-log` flag and keep production runs JSON-only.
2. Create a small shared helper module for capability output parsing so any future API endpoints that shell out to capabilities automatically get robust parsing.
3. Tighten Dashboard error handling:
   - Centralize fetch wrappers to always handle non-OK responses and show clear user messages.
   - Record last successful asset refresh time to help debugging.

## 6. Alignment with CORE_PRINCIPLES

- Functional Simplicity:
  - Introduced a single, focused helper `_parse_script_json` instead of patching each endpoint ad hoc.
- Sequential Logic:
  - Diagnosis → Identify non-JSON prefix → Implement shared parser → Wire into all affected endpoints → Recompile and verify.
- Incremental Testing:
  - Validated changes with `python -m py_compile` and by matching the parser behavior to the actual runner output.
- Zero Assumptions:
  - No longer assume runner `stdout` is pure JSON; we now explicitly handle mixed output.
- Defensive Error Handling:
  - On malformed output, errors are logged with both the exception and raw output to aid future debugging while returning structured HTTP errors to the client.
